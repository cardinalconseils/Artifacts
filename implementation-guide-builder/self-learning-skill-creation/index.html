Framework

How agents learn, create skills,
and improve over time

Not trial-and-error. Four distinct layers — each with a specific job and a specific failure mode. Most implementations only build two.

The core risk: most self-learning setups are garbage-in, garbage-out. The agent learns from its own mistakes without a correction gate. Skills drift. Bad patterns get encoded. You lose control within days.
The four layers

Every self-improving agent needs four layers.

Click each to understand what it does and where it breaks.

DeterministicLayer 1 — Config & routingexpand

YAML rules, routing logic, model assignments. Same input → same decision. This layer never changes unless you change it. It's your control surface.

Hermes: config.yaml. Claude Code: CLAUDE.md + power commands.

Failure mode: too loose. "Any task, any model." The agent improvises. It works until it doesn't.

# config.yaml routing_rules: - task_type: "code_generation" model: "claude-sonnet-4-6" skill_scope: "skills/coding/" - task_type: "research" model: "claude-opus-4-6" skill_scope: "skills/research/"
ProbabilisticLayer 2 — LLM reasoningexpand

The LLM's actual output. Same prompt → different result each run. That's not a bug. The goal is not to make this layer deterministic.

The goal is to constrain its input — what context, skills, and memory it can access — so probabilistic reasoning stays within useful bounds.

Failure mode: trying to fix this with better prompts. Fix Layer 1 (routing) and Layer 3 (memory) instead.

MemoryLayer 3 — Memory & contextexpand

What the agent knows before starting. Three types: short-term (session), long-term (Supabase + Brain MCP), skill memory (reusable procedures queried before similar tasks).

Failure mode: memory grows without curation. Old corrections conflict with new ones. Pruning is not optional.

memory/ agent_memory.md # accumulated facts correction_log.md # mistake → fix pairs skills/ quarantine/ # new, not yet approved validated/ # gatekeeper-approved archived/ # deprecated, for rollback
Skill creationLayer 4 — Skill creation & improvementexpand

New procedures get written, tested, approved, and added to the library. Three triggers: post-task reflection, correction encoding, curriculum ingestion.

Every new skill goes through the Gatekeeper before moving from quarantine/ to validated/. No exceptions.

Failure mode: skipping quarantine. One bad pattern propagates into every future task. You can't trace what broke.

Key insight: layers 1, 3, and 4 are under your control. Layer 2 is not — and doesn't need to be. Design the controllable layers tightly.
Skill anatomy

What a skill actually is

A reusable, versioned procedure — not a prompt. It needs a trigger, a body, validation criteria, and a provenance trail.

01 — Trigger
When to invoke
Keywords or task types that pull this skill into context. Without a precise trigger, skills collide or get missed.
02 — Body
What to do
Step-by-step procedure. Explicit enough that a different agent can execute without ambiguity.
03 — Validation
How to verify
Checklist the output must pass. No criteria = not a skill.
04 — Provenance
Where it came from
Source, version, gatekeeper sign-off, date. Required for audit and rollback.
05 — Scope
Who can use it
Which agents, which task types. Prevents a Python skill contaminating a JS task.
06 — Improvement log
How it changed
Every modification, what triggered it, why. Improvement vs drift.
# skills/validated/python_fastapi_endpoint.md --- trigger: ["create endpoint", "fastapi route"] scope: ["developer_agent"] source: "curriculum: fastapi_course_v2/" version: 3 gatekeeper_approved: "2026-05-20" --- 1. Define Pydantic models for request + response 2. Add route decorator with explicit HTTP method 3. Include dependency injection for auth 4. Write unit test before returning Validation: - [ ] Pydantic model covers all fields - [ ] Auth dependency present - [ ] Test file exists Improvement log: - v1: derived from curriculum - v2: added auth requirement (correction 2026-05-15) - v3: added test requirement (gatekeeper 2026-05-20)

Control architecture

Guardrails that prevent drift

Control what gets permanently encoded. Probabilistic reasoning is fine at runtime. Permanent skill updates need gates.

Without guardrails: the agent improves for three days, one bad correction gets encoded, every subsequent task inherits it. You can't trace it. You start over.
GateWhat it checksFailure action
Format checkHas trigger, body, validation criteria, provenanceStays in quarantine with reason logged
Conflict checkDoesn't contradict an existing validated skillFlag, require manual resolution
Scope checkSkill scope matches the task type it was created forNarrow scope or reject
Source checkProvenance points to vetted source or logged correctionReject — no orphan skills
Human reviewAny skill modifying behavior in a new domainHold in quarantine until reviewed
RollbackVersion history on every skill fileRevert to previous version on flag
Changing behavior

The right way to change how an agent behaves

→
Config or skill update — not a prompt tweak
Update config.yaml or a skill file. Prompts drift and can't be versioned cleanly.
→
Every config change is a commit
Version controlled, commit messages, rollback available. This is your audit trail.
→
Test in quarantine before promoting
Same output as the manual approach → promote. Different → fix the skill body first.
→
Prune stale skills on a cadence
Skills not invoked in 30 days get reviewed. Dead skills accumulate context noise.
Implementation

How to wire this up

Concrete folder structure, config patterns, and the minimum viable setup for self-learning.

project/ config.yaml CLAUDE.md skills/ quarantine/ validated/ archived/ memory/ agent_memory.md correction_log.md user_profile.md curriculum/ source_01/ gatekeeper/ rules.md review_log.md .git/
# Gatekeeper agent prompt Review skill files in skills/quarantine/ against gatekeeper/rules.md. For each skill: 1. Check all required fields are present 2. Search validated/ for trigger conflicts 3. Verify source is traceable 4. Check scope is specific Pass: move to validated/, log approval. Fail: leave in quarantine, log rejection reason. You do not write or modify skills.
CREATE TABLE agent_skills ( id uuid DEFAULT gen_random_uuid() PRIMARY KEY, name text NOT NULL, trigger text[] NOT NULL, scope text[] NOT NULL, body text NOT NULL, validation text NOT NULL, source text NOT NULL, version int DEFAULT 1, status text DEFAULT 'quarantine', approved_at timestamptz, created_at timestamptz DEFAULT now() ); -- Agents read validated only CREATE POLICY agent_read ON agent_skills FOR SELECT USING (status = 'validated');
# Alert triggers - Skill in quarantine > 7 days with no decision - Gatekeeper rejection rate > 40% in a week - Same correction logged 3+ times - Memory file > 50KB - config.yaml modified by an agent # Tools LangSmith → LLM call traces Supabase logs → job state, corrections Git history → config + skill changes Mission Control → Hermes fleet Tmux logs → Claude Code workers
Platform comparison

Hermes v2 vs Claude Code

Same architecture. Different execution. Know what each handles natively and what you build yourself.

Hermes v2
Skill files native — ~/.hermes/skills/
Post-task reflection built in — runs automatically
FTS5 memory search — agent queries past experience
Persistent process — learning accumulates while you sleep
Quarantine is a convention you add — not enforced natively
Gatekeeper wired via YAML sub-agent config
Claude Code
Skill files = power commands in .claude/commands/
Post-task reflection = command you define and trigger
No native cross-session memory — wire Supabase + Brain MCP
Session-bound — learning only persists if you save it
Tmux controller gives direct visibility into every action
Commit hooks enforce rules before patterns land in code
Practical split: Hermes for the persistent learning layer. Claude Code for active dev sprints. Wire both to the same Supabase skills table and Brain MCP. One corpus, two execution modes.
Start here

Minimum viable setup

1
Create the folder structure
skills/quarantine/, validated/, memory/correction_log.md, gatekeeper/rules.md. Under git. Everything builds on this.
2
Write gatekeeper rules first
Define what a valid skill looks like before running any ingestion. The Gatekeeper is only as good as the rules you give it.
3
Run one ingestion end-to-end manually
One small source. Review quarantine yourself. Promote or reject with explicit reasons. Calibrate rules before automating.
4
Add correction log before the feedback loop
Without it, fixes are one-offs. The log is what makes corrections reusable.
5
Automate Gatekeeper only after manual review works
Review the first 20 skill files yourself. Find the gaps. Fix them. Premature automation is how garbage gets in.
