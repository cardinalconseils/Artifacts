Framework

How agents learn, create skills,
and improve over time

Not trial-and-error. A structured architecture with four distinct layers — each with a specific job and a specific failure mode. Most implementations only build two of the four.

The core risk: most self-learning setups are garbage-in, garbage-out. The agent learns from its own mistakes without a correction gate. Skills drift. Bad patterns get encoded. You lose control within days.
The four layers

Every self-improving agent needs four layers.

Click each layer to understand what it does and where it breaks.

Deterministic Layer 1 — Config & routing expand

YAML rules, routing logic, model assignments. Same input → same decision. This layer never changes unless you change it. It's your control surface.

In Hermes: config.yaml — which model handles which task, what tools are available, which skills are in scope.
In Claude Code: CLAUDE.md + power commands — the rules the orchestrator always follows.

Failure mode: the layer is too loose. "Any task, any model" instead of explicit rules. The agent improvises. It works until it doesn't.

# config.yaml — deterministic routing routing_rules: - task_type: "code_generation" model: "claude-sonnet-4-6" skill_scope: "skills/coding/" - task_type: "research" model: "claude-opus-4-6" skill_scope: "skills/research/"
Probabilistic Layer 2 — LLM reasoning expand

The LLM's actual output. Same prompt → slightly different result each run. That's not a bug — it's how language models work. The goal is not to make this layer deterministic.

The goal is to constrain its input — what context, what skills, what memory it can access — so probabilistic reasoning stays within useful bounds.

Failure mode: trying to fix this layer with better prompts. Prompts don't hold. Fix the problem at Layer 1 (routing) and Layer 3 (memory).

Memory Layer 3 — Memory & context expand

What the agent knows before it starts a task. This is where learning actually lives. Three types:

Short-term: current session context.
Long-term: encoded facts, past decisions, correction logs. Persisted across sessions. In Hermes: markdown files in ~/.hermes/memory/. In your stack: Supabase + Brain MCP.
Skill memory: reusable procedures — "how to do X" — derived from past experience. Queried before starting similar tasks.

Failure mode: memory grows without curation. Old corrections conflict with new ones. Regular pruning is not optional.

# Memory structure memory/ agent_memory.md # accumulated facts correction_log.md # mistake → fix pairs user_profile.md # stable constraints skills/ quarantine/ # new skills, not yet approved validated/ # gatekeeper-approved archived/ # deprecated, kept for rollback
Skill creation Layer 4 — Skill creation & improvement expand

New reusable procedures get written, tested, approved, and added to the skill library. Three triggers:

1. Post-task: agent completes a complex task → reflects → writes to skills/quarantine/.
2. Correction: reviewer catches a mistake → fix gets encoded → becomes a skill rule.
3. Curriculum ingestion: you feed a vetted repo or course → agent derives skills from that source.

Every new skill goes through the Gatekeeper before moving from quarantine/ to validated/. No exceptions.

Failure mode: skipping the quarantine step. Skills go live immediately. One bad pattern propagates into every future task. You can't trace what broke.

The key insight: layers 1, 3, and 4 are under your control. Layer 2 is not — and doesn't need to be. Design the controllable layers tightly, and the probabilistic layer stays useful.
Skill anatomy

What a skill actually is

A skill is a reusable, versioned procedure. Not a prompt. Not a system instruction. A procedure with a trigger, a body, validation criteria, and a provenance trail.

01 — Trigger
When to invoke
Keywords or task types that cause the agent to pull this skill into context. Without a precise trigger, skills collide or get missed.
02 — Body
What to do
Step-by-step procedure. Explicit enough that a different agent instance can execute it without ambiguity. Instructions, not prose.
03 — Validation
How to verify
Checklist the output must pass before the skill is considered successful. This is what the Reviewer checks against.
04 — Provenance
Where it came from
Source, version number, gatekeeper sign-off, date. Required for audit and rollback.
05 — Scope
Who can use it
Which agents, which task types. Scoping prevents a skill trained on Python from contaminating a JavaScript task.
06 — Improvement log
How it changed
Every modification, what triggered it, what changed and why. This is how you track improvement vs drift.
# skills/validated/python_fastapi_endpoint.md --- trigger: ["create endpoint", "fastapi route", "REST endpoint"] scope: ["developer_agent"] source: "curriculum: fastapi_course_v2/" version: 3 gatekeeper_approved: "2026-05-20" --- ## Procedure 1. Define Pydantic models for request + response 2. Add route decorator with explicit HTTP method 3. Include dependency injection for auth 4. Add OpenAPI description string 5. Write unit test before returning ## Validation criteria - [ ] Pydantic model covers all fields - [ ] Auth dependency present - [ ] Test file exists at test_[filename].py ## Improvement log - v1: derived from curriculum ingestion - v2: added auth requirement (correction 2026-05-15) - v3: added test requirement (gatekeeper flag 2026-05-20)
Rule: if a skill doesn't have a validation criteria section, it's not a skill — it's a note. The Reviewer needs something concrete to check against.
The core loop

Three ways agents actually learn

Each path creates skills differently. All three route through the same Gatekeeper before anything gets encoded permanently.

Best for: loading a new domain. Give it a course, a codebase, or documentation. It reads, extracts patterns, derives skills. You control what goes in — so you control what comes out.

The agent never invents. It derives from what you fed it. This is the difference between controlled learning and garbage accumulation.
1
Prepare the source
A clean repo, a course folder, a documentation set. Remove anything you don't want the agent learning from. The quality of the input is the ceiling of the output.
2
Run ingestion with explicit scope
Tell the ingestion agent exactly what to extract: patterns, procedures, constraints, anti-patterns. Not "read this and learn" — "read this and extract reusable procedures for [specific task type]."
3
All drafts go to quarantine
Every derived skill lands in skills/quarantine/ — never directly to validated/. The Gatekeeper reviews before anything goes live.
4
Gatekeeper reviews against your rules
Clear trigger? Validation criteria? Conflict with an existing skill? If any check fails, it stays in quarantine with a reason logged.
# Ingestion agent prompt (Hermes or Claude Code) system: | You are an ingestion agent. Read the provided source and extract reusable procedures. Each procedure must follow the skill format exactly. Do not invent. Do not generalize beyond the source. Output to skills/quarantine/ only. source: "./my_course/module_03_api_design/" extract_for: "REST API development tasks" output_path: "skills/quarantine/" max_skills_per_run: 10
Best for: ongoing improvement during active work. Every mistake caught by the Reviewer becomes a learning signal. The agent doesn't just fix the output — it encodes the fix as a skill rule.

Control architecture

Guardrails that prevent drift
without stopping improvement

Control what gets permanently encoded. Probabilistic reasoning is fine at runtime. Permanent skill updates need gates.

Without guardrails: the agent improves for three days, then one bad correction gets encoded. Every subsequent task inherits that bad pattern. You can't trace it. You start over.

Changing behavior

The right way to change how an agent behaves

→
Behavior change = config or skill update, not prompt tweak
If you want the agent to permanently behave differently, update config.yaml or a skill file. Prompts drift and can't be versioned cleanly.
→
Every config change is a commit
Treat config.yaml and skill files like code. Version controlled, commit messages, rollback available. This is your audit trail.
→
Test in quarantine before promoting
Run the new skill against known-good examples before moving it to validated/. Same output as the manual approach → promote. Different → fix the skill body first.
→
Prune stale skills on a cadence
Skills not invoked in 30 days get reviewed. Skills overridden by newer skills get archived. Dead skills accumulate context noise.
Implementation

How to wire this up
in your actual stack

Concrete folder structure, config patterns, and the minimum viable setup for self-learning in both Hermes and Claude Code.

# Minimum viable self-learning structure project/ config.yaml # routing, model assignments, scopes CLAUDE.md # orchestrator rules (Claude Code) skills/ quarantine/ # new skills, not yet approved validated/ # gatekeeper-approved, in active use archived/ # deprecated, kept for rollback memory/ agent_memory.md # accumulated facts + context correction_log.md # mistake → fix pairs user_profile.md # stable preferences + constraints curriculum/ source_01/ # vetted input for ingestion source_02/ gatekeeper/ rules.md # what the gatekeeper checks review_log.md # every decision + reason .git/ # version control on everything above
# gatekeeper/rules.md ## Required fields (all must be present) - trigger: non-empty list of strings - scope: at least one agent name - source: curriculum path or correction_log reference - validation_criteria: at least 2 checkable items - version: integer starting at 1 ## Conflict check - Search validated/ for overlapping triggers - If overlap: flag, do not auto-promote - Resolution: merge, narrow scope, or archive older ## Rejection reasons (must be logged) - MISSING_FIELDS - TRIGGER_CONFLICT - ORPHAN_SOURCE: no traceable provenance - SCOPE_TOO_BROAD: "all agents" is not valid - NO_VALIDATION
# Gatekeeper agent prompt You are the Gatekeeper. Review skill files in skills/quarantine/ against gatekeeper/rules.md. For each skill: 1. Check all required fields are present 2. Search validated/ for trigger conflicts 3. Verify source is traceable 4. Check scope is specific (not "all agents") If all checks pass: move to validated/, log approval. If any check fails: leave in quarantine, append rejection reason to gatekeeper/review_log.md. You do not write skills. You do not modify skills. You only approve, reject, or flag for human review.
-- Supabase: agent_skills CREATE TABLE agent_skills ( id uuid DEFAULT gen_random_uuid() PRIMARY KEY, name text NOT NULL, trigger text[] NOT NULL, scope text[] NOT NULL, body text NOT NULL, validation text NOT NULL, source text NOT NULL, version int DEFAULT 1, status text DEFAULT 'quarantine', approved_at timestamptz, created_at timestamptz DEFAULT now() ); -- Correction log CREATE TABLE correction_log ( id uuid DEFAULT gen_random_uuid() PRIMARY KEY, task_id text, agent_id text, mistake text NOT NULL, correction text NOT NULL, skill_id uuid REFERENCES agent_skills(id), iterations int DEFAULT 1, created_at timestamptz DEFAULT now() ); -- Agents read validated only CREATE POLICY agent_read ON agent_skills FOR SELECT USING (status = 'validated');
# Observability — what to track Every skill created → quarantine entry timestamp Every gatekeeper decision → approve/reject + reason Every correction logged → task_id, mistake, fix Every skill invocation → which skill, which task Skills promoted → quarantine → validated + date Skills archived → when + why # Alert triggers - Skill in quarantine > 7 days, no decision - Gatekeeper rejection rate > 40% in a week - Same correction logged 3+ times (systemic issue) - Memory file > 50KB (needs pruning) - config.yaml modified by an agent (should never happen) # Tools LangSmith → LLM call traces, prompt versions Supabase logs → job state, correction entries Git history → config + skill file changes Mission Control → Hermes fleet visibility Tmux logs → Claude Code worker output
Platform comparison

Self-learning in Hermes vs Claude Code

Same architecture. Different execution. Know what each platform handles natively and what you build yourself.

Hermes v2
Skill files native — markdown in ~/.hermes/skills/
Post-task reflection built in — runs automatically after complex tasks
FTS5 search across memory — agent queries past experience before starting
Persistent process — learning accumulates while you sleep
Quarantine folder is a convention you add — not enforced natively
Gatekeeper wired via YAML sub-agent config
Model swap per agent — one YAML change, no code
Claude Code
Skill files = power commands in .claude/commands/
Post-task reflection is a command you define and trigger
No native cross-session memory — wire Supabase + Brain MCP yourself
Session-bound — learning only persists if you explicitly save it
Tmux controller gives direct visibility into every agent action
Gatekeeper = a dedicated worker in your orchestrator setup
Commit hooks enforce rules before patterns land in the codebase
Practical split: Hermes for the persistent learning layer — curriculum ingestion, always-on memory, background skill improvement. Claude Code for active development sprints where you want direct oversight. Wire both to the same Supabase skills table and Brain MCP. One learning corpus, two execution modes.
Start here

Minimum viable setup

1
Create the folder structure
skills/quarantine/, skills/validated/, memory/correction_log.md, gatekeeper/rules.md. Put it under git. Everything else builds on this.
2
Write your gatekeeper rules first
Before running any ingestion or feedback loops, define what a valid skill looks like. The Gatekeeper is only as good as the rules you give it.
3
Run one curriculum ingestion end-to-end manually
Pick one small, well-scoped source. Run the ingestion agent. Review what lands in quarantine yourself. Promote or reject with explicit reasons. This calibrates your rules before you automate.
4
Add the correction log before adding the feedback loop
Wire the correction log first. Without it, fixes are one-offs and nothing is learned. The log is what makes corrections reusable.
5
Automate the Gatekeeper only after manual review works
Review the first 20 skill files yourself. You'll find gaps in your rules. Fix the rules first, then hand off to the automated Gatekeeper. Premature automation is how garbage gets in.
