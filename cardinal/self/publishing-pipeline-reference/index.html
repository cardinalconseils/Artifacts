Cardinal Conseils — Self publishing

The Cardinal Publishing Pipeline — A Reference Guide

You write a brief on Sunday. You wake Monday knowing it passed validation, was built, and is waiting for your final review — no manual building required.

The four stages in fifteen minutes

WRITE

Output: brief in Obsidian

You compose the brief in Obsidian using the standard template. The format must satisfy the editorial gate — the prompt acts as the only specification.

BRIEF

Gate: frontmatter validation

The cron validates frontmatter and brief completeness. If it passes, the job continues. Otherwise it stays in queue with a log explaining the gap.

BUILD

Artifact: complete index.html

The API builds an artifact from your brief. Same inputs as manual Claude Code but fully automated. Includes validation and download file if requested.

SHIP

Live: auto-PR to main

Files committed as feat/[slug], PR opened with diff and brief. Monday review checks tone, then merge triggers Vercel deploy in href="cdn...">~60 seconds.

Two delivery modes compared

The pipeline supports two speeds, each appropriate at different moments:

Manual — Claude Code now

Run artifact-builder in Claude Code when:

  • The brief is longer than 2,000 tokens — Claude can verify it fits before calling API
  • You want to iterate on tone or layout before publishing
  • The workflow-builder variant is uncertain

Automated — Sunday cron

Use the cron when:

  • The brief is complete and editorial gate passed
  • You finished writing at 11pm and want it built overnight
  • You prefer reviewing finished artifacts rather than building them

What the cron actually does

At 8am EST every Sunday, the job reads queue/ and processes each file in order:

  1. Validates YAML frontmatter with the same rules as manual validation
  2. Detects workflow-builder variant from brief content if not specified
  3. Calls the API with ARTIFACT-BUILD-INSTRUCTIONS.md as system prompt
  4. Writes files to cardinal/{mode}/{slug}/ exactly as manual build
  5. Updates _index.json, llms.txt, root README with new entry
  6. Runs git mv brief processed/ to archive the brief
  7. Opens PR with clear commit history and diff

Failure modes and recovery

Invalid frontmatter

The job dies but logs which field failed. Fix the brief manually and push — cron will retry next Sunday. The artifact subdirectory is never created, so queue/ acts as inbox sorter.

API errors

Rate limits or build errors hit the retry mechanism. If all attempts fail, the brief stays in queue/ with detailed logs. Run process-queue -f [slug] from CLI to retry manually.

Malformed JSON response

API returns unparsable JSON — logged with full prompt/response. Job fails cleanly. Use manual Claude Code with verbose logging() to debug.

Bad PR merge

Deploy step validates _index.json before Vercel build. If merge breaks JSON, Vercel refuses to build. The pipeline fails fast, alerting whichever human triggered the merge.
